# Architecture

**Analysis Date:** 2026-03-21

## Pattern Overview

**Overall:** Multi-application monorepo with two distinct architecture patterns:
1. **Habit Tracker (root)**: Client-side single-page application (SPA) with localStorage persistence and PWA capabilities
2. **Newsletter Digest (newsletter-digest/)**: Backend pipeline with frontend dashboard visualization

**Key Characteristics:**
- Modular component-based architecture for UI applications
- Service-oriented data processing pipeline for backend
- Browser-first data storage with client-side state management
- Scheduled background job execution (Python-based newsletter digest)
- Static data-driven frontend dashboard

## Layers

### Habit Tracker Application

**Presentation Layer:**
- Purpose: Render UI views and handle user interactions
- Location: `js/daily-view.js`, `js/grid-view.js`, `js/habit-editor.js`
- Contains: View rendering functions, DOM manipulation, event handlers
- Depends on: Data layer (data.js), utilities
- Used by: Main application controller (app.js)

**Application Controller:**
- Purpose: Orchestrate view routing, state management, event binding
- Location: `js/app.js`
- Contains: Global state (currentMonthKey, currentDay, currentView), render coordination, lifecycle management
- Depends on: Data layer, all view modules
- Used by: HTML entry point (index.html)

**Data/Model Layer:**
- Purpose: localStorage interface, data transformations, business logic
- Location: `js/data.js`
- Contains: CRUD operations for month data, month key calculations, habit management, check toggles
- Depends on: Browser localStorage API
- Used by: All view modules, app controller

**Service Worker / Offline Support:**
- Purpose: Cache assets, enable offline functionality, PWA installation
- Location: `service-worker.js`
- Contains: Cache-first strategy, asset pre-caching, lifecycle management
- Depends on: Fetch API, Cache API
- Used by: Browser runtime

**Styling:**
- Location: `css/styles.css`
- Contains: Layout, typography, modal styling, responsive design
- Includes: Daily view, grid view, modal components

### Newsletter Digest Pipeline

**Data Extraction:**
- Purpose: Collect content from multiple sources
- Location: `newsletter-digest/gmail_client.py`, `newsletter-digest/rss_fetcher.py`
- Contains: Email fetching, RSS parsing, date filtering
- Depends on: Gmail API, HTTP feeds
- Used by: Summarizer

**Content Detection & Processing:**
- Purpose: Filter and classify newsletter content
- Location: `newsletter-digest/newsletter_detector.py`, `newsletter-digest/email_parser.py`
- Contains: Newsletter detection logic, sender matching, keyword filtering
- Depends on: Data extraction, configuration
- Used by: Summarizer

**Summarization & Trend Extraction:**
- Purpose: Generate AI-powered summaries and structured trend data
- Location: `newsletter-digest/summarizer.py`, `newsletter-digest/trend_extractor.py`
- Contains: Claude API calls, markdown formatting, trend classification
- Depends on: Anthropic SDK, configuration
- Used by: Formatter and dashboard

**Formatting & Distribution:**
- Purpose: Convert processed content to consumable formats
- Location: `newsletter-digest/digest_formatter.py`, `newsletter-digest/gmail_client.py`
- Contains: HTML email generation, image optimization, send logic
- Depends on: Gmail API, formatters
- Used by: Main entry point

**Configuration & Utilities:**
- Purpose: Centralize settings and helpers
- Location: `newsletter-digest/config_loader.py`
- Contains: YAML parsing, API credentials, feed list management
- Depends on: Environment variables, file system
- Used by: All modules

**Dashboard Data:**
- Purpose: Serve structured trend data for visualization
- Location: `trends-dashboard/data/` (JSON files generated by trend_extractor)
- Contains: Weekly digests with structured topic, speaker, trend metadata
- Generated by: trend_extractor.py
- Used by: Dashboard frontend

## Data Flow

### Habit Tracker - Daily Use

1. **Initialization** (DOMContentLoaded in app.js)
   - Load configuration and today's info from data.js
   - Create/fetch current month data
   - Bind event listeners
   - Render initial view

2. **User Action** (e.g., toggle habit check)
   - Event fires on checklist item
   - daily-view.js calls toggleCheck() from data.js
   - data.js updates localStorage
   - View updates optimistically (toggle CSS class)
   - Progress bar recalculates

3. **View Navigation**
   - User clicks month arrow or date nav button
   - app.js updates currentMonthKey/currentDay state
   - app.js calls render() which dispatches to appropriate view renderer
   - View renderer pulls fresh data from data.js
   - New view renders into container

4. **Habit Management**
   - User clicks edit button
   - habit-editor.js opens modal
   - Modal loads habits from current month
   - User saves changes via updateHabits()
   - app.js re-renders active view with new habit list

### Newsletter Digest Pipeline

1. **Configuration Load**
   - main.py loads config.yaml via config_loader
   - Reads API keys from environment, feed list from config

2. **Email & RSS Fetch** (Parallel)
   - gmail_client.fetch_emails() queries Gmail API with date filter
   - rss_fetcher.fetch_feeds() fetches and parses all configured RSS feeds
   - Both filtered by lookback_days window

3. **Content Classification**
   - newsletter_detector.detect_newsletters() marks emails as newsletters
   - Applies sender whitelist and keyword matching
   - Filters irrelevant content

4. **Summarization**
   - summarizer.summarize() sends newsletters + RSS articles to Claude
   - Returns markdown summary grouped by topic

5. **Trend Extraction** (Parallel)
   - trend_extractor.extract_trends() calls Claude again for structured data
   - Produces JSON with topics, speakers, trend metadata
   - Writes to trends-dashboard/data/ directory

6. **Dashboard Data Push**
   - push_dashboard_data() commits/pushes new JSON files to GitHub
   - GitHub Pages serves updated dashboard automatically

7. **Email Distribution** (Optional)
   - digest_formatter.format_digest_html() creates styled HTML
   - gmail_client.send_email() delivers to recipient via Gmail API

### State Management

**Habit Tracker:**
- **Client-side only**: Single page state lives in app.js (currentMonthKey, currentDay, currentView)
- **Persistent**: All data lives in localStorage, survives page reload
- **Month data structure**:
  ```
  {
    month: "2026-03",
    habits: [{ id, name, order }, ...],
    checks: { habitId: [dayNumbers], ... }
  }
  ```

**Newsletter Digest:**
- **No persistent state**: Each run starts fresh (stateless pipeline)
- **Credentials**: Gmail auth via credentials.json + token.json (OAuth flow)
- **Config**: YAML-driven configuration with overrides

**Trends Dashboard:**
- **Client-side filtering**: Global state tracks selectedTopic filter
- **Data source**: Static JSON files generated offline
- **No backend**: Pure static site served via GitHub Pages

## Key Abstractions

**Month Key (YYYY-MM format):**
- Purpose: Canonicalize date references for data organization
- Examples: `toMonthKey()`, `parseMonthKey()`, `offsetMonth()` in data.js
- Pattern: String-based key enables easy sorting and localStorage namespacing

**Check Array:**
- Purpose: Store which days (1-31) a habit was completed
- Examples: `checks[habitId] = [1, 3, 5, 7, ...]` (sorted array of day numbers)
- Pattern: Sparse array avoids storing false values

**Habit Object:**
- Purpose: Define a tracked habit
- Structure: `{ id: string, name: string, order: number }`
- Pattern: Immutable-style updates via `.map()` to preserve ordering

**Week Digest (Dashboard):**
- Purpose: Organize trends data by calendar week
- Structure: JSON with topics, articles, speakers, metrics
- Pattern: Generated by trend_extractor, consumed by chart renderers

## Entry Points

**Habit Tracker:**
- Location: `/index.html`
- Triggers: Browser requests root URL
- Responsibilities: Bootstrap SPA, load service worker, mount app.js

**Newsletter Digest:**
- Location: `newsletter-digest/main.py`
- Triggers: Scheduled cron job or manual CLI invocation
- Responsibilities: Orchestrate pipeline, handle dry-run/trends-only modes, push to dashboard

**Trends Dashboard:**
- Location: `trends-dashboard/index.html`
- Triggers: Browser requests dashboard URL
- Responsibilities: Load data, initialize state, render all sections

## Error Handling

**Strategy:** Graceful degradation with user feedback

**Patterns:**

- **View Rendering**: Empty state UI if no data exists (habit tracker welcome screen)
- **localStorage Fallback**: If data missing, create from prior month (habit inheritance)
- **API Failures**: Try/catch in fetch_emails/fetch_feeds with warnings
- **Missing Days**: Default to day 1 when navigating between months
- **Duplicate Habits**: Alert user on save if duplicates detected
- **Service Worker**: Catch errors silently, fall back to network if cache miss

## Cross-Cutting Concerns

**Logging:**
- Habit Tracker: None (console only, via browser DevTools)
- Newsletter Digest: Print statements to stdout (CLI feedback)
- Dashboard: Browser console only

**Validation:**
- Habit names: Max 30 chars, trimmed, deduplicated (habit-editor.js)
- Month navigation: Clamped to 1-31 days, month overflow handled
- Check toggles: Idempotent (add/remove same day multiple times = safe)
- RSS feeds: Date filtering by published date, lookback days window

**Authentication:**
- Habit Tracker: None (client-side only, localStorage)
- Newsletter Digest: OAuth2 via Gmail API (credentials.json + token.json)
- Dashboard: None (public static site)

**Security:**
- Habit Tracker: HTML escaping in all views (escapeHtml, escapeAttr functions)
- Newsletter Digest: Email parsing sanitized, no untrusted code execution
- Dashboard: No user input, no XSS vectors

**Date Handling:**
- All dates stored as month keys (YYYY-MM) or day numbers (1-31)
- formatMonthLabel() and formatDayLabel() handle localization
- todayInfo() captures local date at runtime
- No timezone concerns (local browser time assumed)
